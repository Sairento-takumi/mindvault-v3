- **작성일**: 2026-05-30
- **상태**: Draft (사용자 검토 대기)
- **범위**: 마스터 로드맵(중장기 비전) + Phase 1 상세 설계
- **현재 기준 버전**: v3.4.1 (`eda596e`)

## 0. 한 줄 요약

MindVault를 *세션 연속성 장기기억*에서 *개인 지식 자산화 세컨드 브레인*으로 진화시킨다. 단, **신뢰 가능한 효과적 회수(토대)를 먼저 못 박은 뒤** 자산화 기능을 얹는다 — 토대 없이 확장하면 v1 폐기(노이즈 주입) 패턴이 더 큰 스케일로 재현된다.

---

## 1. 배경 & 문제 정의

### 1.1 계기
2026-05-30 외부 영상(커리어해커 알렉스, 메타) 분석에서 출발. 영상의 "세컨 브레인"(Karpathy LLM Wiki 패턴)과 MindVault를 비교한 결과, **둘은 더 낫고 못함의 관계가 아니라 목적이 다름**:
- 알렉스 = 개인 지식 *자산화* (소비·생산한 지식을 영구 자산으로, 질의응답·생성에 활용)
- MindVault v3 = Claude Code *세션 연속성* (새 세션 딴소리 방지)

사용자 판단: MindVault의 중장기 지향점은 후자에서 전자로 넓어져야 한다. 단순 세션 연속성보다 자산화 세컨드 브레인의 활용성이 높기 때문.

### 1.2 토대의 결함 (실측 근거)
자산화로 확장하기 전에, 현재 v3가 약속한 "장기기억"조차 효과적으로 작동하지 않는다는 실측 증거가 있다.

- **회수 통합률(strict cited) = 7.62% baseline** (NEXT-37 측정, `src/self_eval.py` `recall_utilization`/`classify_recall_utilization`). 회수는 되지만 답변 reasoning에 거의 통합되지 않음.
- 두 가지 반대 방향 실패 모드가 같은 뿌리(출처 없는 "주장")에서 나옴:
  - **under-integration**: 회수해도 답변에 안 씀 (7.62%). [[recalled-memory-weight]]의 원래 문제.
  - **over-trust**: 회수한 stale 메모리를 검증 없이 믿음. 본 로드맵을 만든 2026-05-30 대화 자체에서 재현됨 — 어시스턴트가 stale한 "BGE-M3" 임베딩 표기를 믿고 시스템 상태를 틀리게 답함(실제는 Arctic-ko v2.0, Sprint 14 교체).

사용자 명시(2026-05-27): "v4 진입 시 해결할 문제가 아니라 지금 당장 해결할 문제다. 장기기억으로서의 v3조차 제대로 작동하지 않고 있다는 증거니까."

### 1.3 사용자가 강조한 원칙
- **메모리는 100% 신뢰 대상이 아니다.** 사람의 기억처럼 시간·상황에 따라 흔들린다(오히려 더 심함). 따라서 비전은 휘발성 메모리 한 줄이 아니라 검증된 영속 설계 문서(본 문서)에 닻을 둔다.
- 진단·설계는 메모리 추정이 아니라 실측(grep/코드)에 근거한다.

## 2. 비전 & 최종상

### 2.1 비전
형이 접하고 (형+Claude가) 함께 도달한 지식을 **신뢰 가능한 형태로 영구 자산화**하고, 질의응답·개념 간 통찰 생성에 활용하는 개인 세컨드 브레인.

- 중심축: **개인 지식 자산화**
- 입력 소스 최종 범위: **Claude Code 세션·대화·판단 + 형이 던지는 외부 콘텐츠(영상·기사·문서) ingest**
- 명시적 비범위: 형 생산물(유튜브 대본·코드·블로그) 자동 수집 — 외부 연동이 필요해 "CC 내부 전용" 제약과 충돌하므로 제외

### 2.2 최종상 — 검증 가능한 정의 (yes/no 판정)
"세컨드 브레인이 됐다"의 합격 기준:
- **ⓐ** 형이 던진 외부 콘텐츠가 자동 distill → 기존 지식과 연결된다
- **ⓑ** 형이 없을 때 Claude가 형의 과거 판단을 **출처와 함께** 답한다
- **ⓒ** 회수된 기억이 답변 reasoning에 실제 통합된다 (recall 통합률이 baseline 7.62%에서 유의하게 상승)
- **ⓓ** 형이 메모리를 "믿을 수 있다" (출처 검증 가능 + stale/모순 자동 정리)

## 3. 로드맵 (토대 우선)

각 Phase는 검증 가능한 완료 게이트로 닫고, 게이트를 통과해야 다음으로 간다(빅뱅 금지).

| Phase | 이름 | 핵심 | 완료 게이트 |
|---|---|---|---|
| 0 | 현재 (v3.4.1) | 세션 연속성 4-layer + 모순감지 | (작동 중, 단 ⓒ·ⓓ 결함) |
| **1** | **토대: 신뢰 가능한 효과적 회수** | provenance(출처) + 효과적 회수 강화 + 신뢰성 검증 | 신규 영구기억 출처 100% + recall strict cited 상승(dogfood 정량) + stale 회귀 차단 |
| 2 | 지식 구조 | 타입 엣지(supports/extends/refutes/depends_on) + 개념 롤업 페이지 | 회수가 근거/반박 동반, 개념 허브 자동 생성 |
| 3 | 외부 ingest | URL 던지면 distill→기존 지식 연결 (`/benchmark` 자산 재활용) | 영상 1편 던져 자산화→질의응답 통과 (ⓐ) |
| 4 | 능동 활용 | 개념 간 통찰 생성 + 보이스/스타일 | 형 보이스 초안을 손 안 대고 사용 |

**순서 근거 (토대 우선 채택)**: 신뢰성·효과적 회수가 안 되는 상태에서 외부 ingest를 늘리면 노이즈가 증폭되고 v1 폐기 패턴이 재현된다. 구조(Phase 2)를 먼저 깔아야 ingest(Phase 3)된 지식이 연결될 자리가 생긴다.

## 4. Phase 1 상세 — 토대: 신뢰 가능한 효과적 회수

### 4.1 목표
회수가 "출처 없는 주장"이라 반영도 검증도 안 되는 뿌리를 해소한다. 두 실패 모드(under-integration / over-trust)를 함께 다룬다.

### 4.2 컴포넌트 (3축)

**① Provenance (출처) — 뿌리 해결**
- 영구기억 frontmatter에 출처 보존: `source: {type: session|commit|url, ref, ts}`
- 신규 메모리는 추출 시 자동 기록. 기존 171개는 점진 백필(선택적, 코드/사실 참조 메모리 우선).
- 회수 시 **짧은 출처 라벨**을 동반 주입한다. 원문 전체는 필요할 때만 Read → 토큰낭비 0(v1 교훈 준수).
- 효과: 출처가 있으면 (a) 반영 동기가 생기고(under-integration 완화) (b) 검증이 가능해진다(over-trust 완화). 두 실패를 한 메커니즘으로 누른다.

**② 효과적 회수 강화 (under-integration)**
- 현재 Chain-of-Note "회수 노트:" 한 줄(NEXT-37)은 약한 강제. 여기에 self-check 계약을 추가.
- self-check 계약: **옵션/권장/다음 단계 제시 직전**, 회수된 feedback·project 메모리의 명시 룰과 cross-reference. 룰을 위반하는 옵션은 제거하거나 "회수 메모리 X 위반 가능성"으로 명시 표기. ([[recalled-memory-weight]]의 How-to-apply를 계약으로 승격)
- 측정: `self_eval` strict cited %를 추적하고 목표치 설정 후 1주 dogfood로 정량 개선 확인.

**③ 신뢰성 검증 (over-trust)**
- stale 자동 감지: 코드/사실 참조를 담은 메모리(file:line, 모델명, 버전 등)에 재검증 플래그를 부여하고 주기적으로 현행 코드와 대조.
- 2026-05-30 BGE→Arctic stale 사고를 회귀 테스트 케이스로 고정한다.

### 4.3 완료 게이트 (검증 가능)
1. 신규 영구기억의 출처 추적률 100%
2. recall strict cited: baseline 7.62% → 목표치(dogfood 정량 개선 확인). 목표 수치는 Phase 1 구현 계획에서 확정.
3. stale 감지가 BGE류 사고를 1건 이상 사전 차단(회귀 케이스 통과)

### 4.4 솔직한 한계 (의도적 설계 선택)
**회수 통합은 결국 메인 Claude의 행동이라 hook이 100% 강제할 수 없다.** 시스템이 보장하는 범위는 *주입 + 출처 + 측정 + self-check 계약*까지이고, 실제 사용은 모델 몫이다.

따라서 Phase 1은 "100% 통합"을 약속하지 않는다. 목표는 **출처로 신뢰를 높여 자발적 사용을 유도 + 측정으로 압박 + self-check 계약**까지다. 자동 게이트 조정은 여전히 미구현(`self_eval` 철학 유지 — 잘못 학습된 loop가 게이트를 망가뜨릴 위험이 더 큼).

## 5. 신뢰성 원칙 (전 Phase 공통)

- 모든 영구기억에 출처(session/commit/url) + 타임스탬프 보존, 회수 시 출처 라벨 동반.
- 메모리는 "주장"이 아니라 "출처 있는 근거"로 다룬다.
- 코드/사실 참조 메모리는 주기적 재검증(stale 차단).
- 모순감지(Layer 5) 유지·강화.
